# Gauri Evaluation Pipeline

This folder provides a lightweight evaluation dashboard for the existing Gauri
Problem 1 and Problem 2 pipelines.

It intentionally reports three different metric types separately:

| Metric type | Meaning |
| --- | --- |
| Problem 1 internal quality | Automated coverage and consistency checks. These are not verified accuracy. |
| Problem 1 human-reviewed pilot accuracy | Accuracy calculated only from completed reviewer verdicts in the review queue. |
| Problem 1 LLM-assisted silver estimate | Optional research-assisted estimate from `llm_*` verdict columns. This is not golden-set accuracy. |
| Problem 2 facade proxy agreement | Agreement with the existing sign and entrance proxy benchmark. This is not human-verified facade accuracy. |

## Run

From the `Gauri` folder:

```bash
python3 Evaluation_pipeline/src/evaluate.py
```

The script uses only the Python standard library.

To include an LLM-researched silver dataset:

```bash
python3 Evaluation_pipeline/src/evaluate.py \
  --llm-silver-input /path/to/problem1_review_queue_llm_verified.csv
```

## Inputs

| Input | Source |
| --- | --- |
| `Conflation_pipeline/outputs/final_problem1_conflation.csv` | Generated by the Problem 1 pipeline. |
| `Problem2_facade_pipeline/benchmark_dataset/problem2_proxy_evaluated_building_based_rows.csv` | Existing Problem 2 sign and entrance proxy evaluation rows. |

Problem 2 metrics are omitted if the proxy benchmark CSV is absent.

## Outputs

Outputs are written under `Evaluation_pipeline/outputs/`:

| Output | Purpose |
| --- | --- |
| `evaluation_summary.csv` | Concise dashboard for demos and progress tracking. |
| `problem1_quality_metrics.csv` | Full Problem 1 automated quality report. |
| `problem1_automated_flags.csv` | POIs with suspicious or review-worthy patterns. |
| `problem1_review_queue.csv` | Balanced challenge sample for finding failure modes. |
| `problem1_random_review_queue.csv` | Simple random sample for estimating overall performance. |
| `problem1_reviewed_metrics.csv` | Human-reviewed pilot metrics calculated from completed review verdicts. |
| `problem1_llm_silver_metrics.csv` | Optional LLM-assisted estimates, kept separate from human-reviewed accuracy. |
| `problem1_llm_silver_review.csv` | Local copy of the supplied LLM-researched CSV for audit and human approval. |
| `problem2_proxy_metrics.csv` | Problem 2 facade proxy benchmark metrics. |

## Problem 1 Internal Quality Score

Each building-relevant POI can receive up to 100 points:

| Evidence | Points |
| --- | ---: |
| Matched address exists | 20 |
| Matched building exists | 20 |
| POI and matched address support the same building | 30 |
| Address matching confidence is high or medium | 15 |
| A street was extracted from the matched address | 10 |
| Nearest street segment name agrees with the matched street | 5 |

This score is useful for comparing pipeline versions. It must not be presented
as real-world accuracy.

## Build A Golden Dataset

Open `problem1_review_queue.csv` in a spreadsheet. For each sampled POI, inspect
open and permitted evidence sources, then fill these fields:

```text
review_address_verdict
review_building_verdict
review_street_verdict
review_poi_class_verdict
review_overall_verdict
evidence_source_urls
review_notes
reviewed_by
reviewed_at
```

Allowed verdicts:

```text
correct
incorrect
ambiguous
uncertain
not_applicable
```

Run the evaluator again after adding verdicts. Existing reviewer fields are
preserved when the queue is regenerated. Accuracy denominators include only
`correct` and `incorrect` verdicts; ambiguous and uncertain rows remain visible
but do not force a misleading answer.

## LLM-Assisted Silver Labels

An internet-searching LLM can accelerate research, but its output is not a
golden dataset. Supply the enriched CSV with `--llm-silver-input`. The evaluator
reports those estimates under `problem1_llm_silver` and stores a local copy for
audit. Review and approve the rows manually before copying verdicts into the
`review_*` columns used for human-reviewed accuracy.

## Suggested First Review

Start with the default 50-row queue. It intentionally includes strong matches,
weak matches, conflicts, and non-building POIs. For a larger sample:

```bash
python3 Evaluation_pipeline/src/evaluate.py --review-sample-size 100
```

The evaluator also generates a reproducible 50-row simple random sample. Use
that file when estimating overall performance. The balanced queue is better for
debugging because it deliberately over-represents difficult rows.

To change the random sample size:

```bash
python3 Evaluation_pipeline/src/evaluate.py --random-review-sample-size 100
```
